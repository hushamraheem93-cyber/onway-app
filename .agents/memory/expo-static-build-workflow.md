---
name: Expo static build workflow reliability
description: Why the "Start Backend" workflow intermittently fails to produce Expo Go manifests, and how it was fixed.
---

This project serves Expo Go manifests from a pre-built static snapshot (`static-build/{platform}/manifest.json`), generated by `scripts/build.js` (never edit that script — it's under the forbidden `scripts/` directory). The workflow command chains three jobs with `&`: build+start the Express server, wait for Metro health then run the static build, and run Metro itself in the foreground.

**Problem:** without a signal trap, `SIGTERM` sent on workflow restart only kills the foreground process. Backgrounded jobs (`server:prod`, the "wait for metro then build" subshell) survive as orphans. The next start spawns a fresh set, so two `scripts/build.js` instances can run concurrently, racing on the same `static-build/` directory (one process's `prepareDirectories()` wipes files the other just wrote) — this silently corrupts the build, leaving `manifest.json` missing/empty even though logs show "ios manifest ready".

**Fix:** prefix the workflow command with `trap 'kill 0' EXIT SIGTERM SIGINT;` so any restart signal propagates to the whole process group and cleans up every backgrounded job before new ones start.

**How to apply:** if Expo Go / manifest 404s reappear after a workflow restart, first check for duplicate `scripts/build.js`, `expo start`, or `server_dist/index.js` processes (`ps aux`) and stale listeners on 5000/8081/8082 before assuming a code-level bug. Kill stray PIDs, clear `static-build/`, and restart cleanly.
