---
name: Expo static build workflow reliability
description: Why the "Start Backend" workflow intermittently fails to produce Expo Go manifests, and how it was fixed.
---

This project serves Expo Go manifests from a pre-built static snapshot (`static-build/{platform}/manifest.json`), generated by `scripts/build.js` (never edit that script — it's under the forbidden `scripts/` directory). The workflow command chains three jobs with `&`: build+start the Express server, wait for Metro health then run the static build, and run Metro itself in the foreground.

**Problem:** without a signal trap, `SIGTERM` sent on workflow restart only kills the foreground process. Backgrounded jobs (`server:prod`, the "wait for metro then build" subshell) survive as orphans. The next start spawns a fresh set, so two `scripts/build.js` instances can run concurrently, racing on the same `static-build/` directory (one process's `prepareDirectories()` wipes files the other just wrote) — this silently corrupts the build, leaving `manifest.json` missing/empty even though logs show "ios manifest ready".

**Fix (partial):** prefix the workflow command with `trap 'kill 0' EXIT SIGTERM SIGINT;` so a restart signal propagates to the whole process group and cleans up every backgrounded job before new ones start. This alone was NOT sufficient in practice — orphans still survived across restarts (observed repeatedly: duplicate `expo start`/`server_dist`/`build.js` process trees, sometimes 10+ stacked instances, Metro hitting the interactive "Port 8081 is being used, use 8082 instead?" prompt and hanging forever, `expo:static:build` failing with `ENOTEMPTY` on `static-build/`).

**Fix (robust):** additionally prefix the command with `fuser -k 5000/tcp 8081/tcp 8082/tcp 2>/dev/null; sleep 1; rm -rf static-build;` before starting the build/server/Metro chain, so every restart forcibly frees the ports and wipes any half-written build directory regardless of what orphans exist from prior sessions. Full command used:
`trap 'kill 0' EXIT SIGTERM SIGINT; fuser -k 5000/tcp 8081/tcp 8082/tcp 2>/dev/null; sleep 1; rm -rf static-build; npm run server:build && npm run server:prod & (until curl -sf http://localhost:8081/status >/dev/null 2>&1; do sleep 2; done && npm run expo:static:build) & npm run expo:dev`

**How to apply:** if Expo Go / manifest 404s or "can't open the app" reports reappear, first check for duplicate `scripts/build.js`, `expo start`, or `server_dist/index.js` processes (`ps -eo pid,ppid,cmd | grep -iE "expo start|server_dist|scripts/build"`) and stale listeners on 5000/8081/8082 before assuming a code-level bug. Kill stray PIDs by exact PID (broad `pkill -f` can kill the invoking shell itself in this sandbox), clear `static-build/`, and restart. If it recurs even with the fuser-based command in place, suspect that some orphan process predates the current workflow config entirely (started manually or by a much earlier session) — hunt it down by port with `ss -tlnp` / `lsof -i` rather than assuming the workflow command is at fault.
