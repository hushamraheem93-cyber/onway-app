أريد إنشاء نظام اختبار احترافي لتطبيق OnWay يغطي جميع أجزاء النظام بالكامل قبل النشر.

يشمل الاختبار:

1- اختبار تطبيق الزبون (Customer App)

* إنشاء مستخدمين تجريبيين تلقائياً.
* تسجيل الدخول والخروج.
* البحث عن المنتجات.
* إضافة المنتجات للسلة.
* تعديل الكميات.
* حذف المنتجات.
* اختيار العنوان.
* إنشاء الطلب.
* الدفع (محاكاة).
* إلغاء الطلب.
* تتبع الطلب حتى التسليم.
* اختبار الإشعارات.
* اختبار المفضلة.
* اختبار التقييمات.

2- اختبار تطبيق التاجر (Merchant App)

* إنشاء متجر تجريبي.
* رفع مئات المنتجات تلقائياً.
* تعديل المنتجات.
* حذف المنتجات.
* تغيير الأسعار.
* تغيير حالة المنتج (متوفر / نفد).
* استقبال الطلبات.
* قبول الطلب.
* رفض الطلب.
* بدء التحضير.
* إنهاء التحضير.
* انتظار السائق.
* مراجعة التقارير.
* اختبار لوحة التحكم بالكامل.

3- اختبار تطبيق السائق (Driver App)

* إنشاء عشرات السائقين التجريبيين.
* تسجيل الدخول.
* استلام الطلب.
* قبول الطلب.
* رفض الطلب.
* الوصول للمتجر.
* تأكيد استلام الطلب.
* بدء التوصيل.
* الوصول للزبون.
* تسليم الطلب.
* إنهاء المهمة.
* اختبار تحديث الموقع GPS كل عدة ثوان.
* اختبار الإشعارات.

4- اختبار لوحة الإدارة (Admin Dashboard)

* تسجيل الدخول.
* إدارة المستخدمين.
* إدارة السائقين.
* إدارة التجار.
* إدارة المنتجات.
* إدارة الطلبات.
* إدارة العمولات.
* إدارة الإشعارات.
* إدارة الكوبونات.
* إدارة المحافظ.
* إدارة التقارير.
* اختبار جميع الصلاحيات.

5- اختبار دورة الطلب كاملة (End-to-End)
محاكاة السيناريو التالي آلاف المرات:
الزبون ينشئ طلب →
يصل للتاجر →
يوافق التاجر →
يبدأ التحضير →
السائق يقبل المهمة →
السائق يستلم الطلب →
السائق يسلمه →
يتغير الطلب إلى مكتمل →
تحديث أرباح السائق →
تحديث أرباح التاجر →
تحديث تقارير الإدارة.

6- Load Testing
محاكاة:

* 100 مستخدم.
* 500 مستخدم.
* 1000 مستخدم.
* 3000 مستخدم.
* 5000 مستخدم.
* 10000 مستخدم متزامن.

مع تنفيذ:

* إنشاء الطلبات.
* البحث.
* رفع المنتجات.
* تحديث الطلبات.
* الإشعارات.
* الخرائط.
* تتبع السائق.

7- Stress Testing
زيادة الحمل حتى نقطة الانهيار لمعرفة:

* الحد الأقصى للطلبات بالثانية.
* الحد الأقصى للمستخدمين.
* استهلاك CPU.
* استهلاك RAM.
* استهلاك قاعدة البيانات.
* سرعة API.
* زمن الاستجابة.
* معدل الأخطاء.

8- Database Testing
اختبار:

* إنشاء البيانات.
* التعديل.
* الحذف.
* المعاملات Transaction.
* العلاقات.
* سلامة البيانات.
* عدم تكرار الطلبات.
* عدم فقدان البيانات.

9- Security Testing
اختبار:

* JWT.
* Authentication.
* Authorization.
* SQL Injection.
* XSS.
* CSRF.
* Rate Limiting.
* صلاحيات المستخدم.
* صلاحيات السائق.
* صلاحيات التاجر.
* صلاحيات الإدارة.

10- Performance Report
بعد انتهاء جميع الاختبارات يتم إنشاء تقرير يحتوي على:

* عدد الاختبارات المنفذة.
* عدد الاختبارات الناجحة.
* عدد الاختبارات الفاشلة.
* أسباب الفشل.
* زمن الاستجابة لكل API.
* استهلاك الخادم.
* استهلاك قاعدة البيانات.
* عدد الطلبات بالثانية (RPS).
* أقصى عدد مستخدمين متزامنين.
* التوصيات لتحسين الأداء.

11- أدوات الاختبار
استخدم أفضل الأدوات المناسبة للمشروع مثل:

* Playwright أو Flutter Integration Test لاختبارات الواجهة.
* k6 أو JMeter لاختبارات الضغط.
* Postman/Newman أو REST Assured لاختبارات الـ API.
* اختبارات تعمل تلقائياً ويمكن تشغيلها بضغطة زر قبل كل إصدار جديد.

يجب أن تكون جميع الاختبارات قابلة للتشغيل تلقائياً (Automated Testing)، مع إنشاء بيانات تجريبية وتنظيفها بعد انتهاء الاختبار، وإصدار تقرير نهائي يوضح جاهزية النظام للإطل
