📐 تحليل المعمارية — كيف يعمل التطبيق مع Firestore

نتيجة: آمنة
✅
جميع العمليات تمر عبر Express Backend — لا قراءة مباشرة
فُحصت جميع ملفات client/ — لا يوجد أي استخدام مباشر لـ Firestore SDK (لا getDocs ولا onSnapshot ولا setDoc). البيانات تُقرأ وتُكتب حصراً عبر REST API على المنفذ 5000.
✅
Firebase Admin SDK يتجاوز القواعد — وهذا مقصود
الخادم يستخدم Admin SDK الذي يتجاوز قواعد Firestore. القواعد تعمل كطبقة دفاع ثانية ضد أي وصول مباشر محتمل من العميل أو من مهاجم استخدم مفاتيح Firebase العامة لكتابة استعلامات مباشرة.
ℹ️
client/lib/firebase.ts — مُهيّأ لكنه غير مُستخدم للقراءة المباشرة
الملف يُهيّئ Firebase app ويُصدّر db، لكن لا يوجد أي ملف آخر في client/ يستورد هذا الـ db ويستخدمه للاستعلام. المفتاح العام (apiKey) في هذا الملف مقبول — Firebase يُصمّمه ليكون عاماً.

🔍 نتائج المراجعة الأمنية

0 ثغرات حرجة
✅
لا يوجد allow read, write: if true في أي مكان
فُحص الملف كاملاً — لا توجد قاعدة خطيرة تسمح بوصول كامل. القاعدة الافتراضية match /{'{'}document=**{'}'} تمنع أي مجموعة غير مصرّح بها.
✅
بيانات المستخدمين (هاتف، عنوان، طلبات) محمية بالكامل
users وaddresses وorders — مرفوض الوصول إليها كلياً من العميل. لا يمكن لأي مستخدم قراءة بيانات مستخدم آخر حتى لو عرف معرّفه.
✅
البيانات المالية للسائقين محمية
driverWallets وwalletHistory — محمية بالكامل. لا يمكن لأي طرف قراءة أرصدة السائقين أو معاملاتهم مباشرةً.
✅
أكواد الترويج محمية من الاستغلال المباشر
promoCodes وpromoUsage — مرفوضة. لا يمكن لأي مستخدم استعراض الأكواد المتاحة أو التحايل على منطق التحقق من الكود.
✅
رموز Push Tokens محمية من الحصاد
pushTokens — مرفوضة. لا يمكن لجهة خارجية جمع رموز الإشعارات وإرسال إشعارات مزيفة للمستخدمين.
✅
القاعدة الافتراضية تحمي المجموعات المستقبلية تلقائياً
إذا أُضيفت مجموعة Firestore جديدة في المستقبل دون تحديث القواعد، ستكون محمية تلقائياً بقاعدة الرفض الافتراضية.

🚀 نشر القواعد على Firebase

إجراء مطلوب قبل الإطلاق

ملف firestore.rules جاهز في المشروع. لتفعيله على Firebase:

الطريقة 1 — Firebase CLI (موصى بها)
npm install -g firebase-tools
firebase login
firebase deploy --only firestore:rules
الطريقة 2 — يدوياً عبر Firebase Console
  1. افتح console.firebase.google.com ← Firestore ← Rules
  2. انسخ محتوى ملف firestore.rules من المشروع
  3. الصقه في المحرر واضغط Publish
⚠️
القواعد الحالية في Firebase Console قد تكون مختلفة
حتى يتم نشر الملف الجديد، القواعد الحالية في Firebase لا تزال سارية. يجب النشر قبل الإطلاق الإنتاجي.

💾 تفعيل النسخ الاحتياطي اليومي لـ Firestore

لا يمكن تفعيله برمجياً — خطوات يدوية

تفعيل النسخ الاحتياطي يتطلب خطة Blaze (Pay-as-you-go) في Firebase. إليك الخطوات:

  1. افتح Firebase Console للمشروع
    اذهب إلى console.firebase.google.com ← onway-74c20 ← Firestore Database
  2. افتح تبويب "Backups"
    في القائمة الجانبية لـ Firestore، ابحث عن تبويب "Backups" أو "Scheduled backups".
  3. أنشئ جدول نسخ احتياطي جديد
    اضغط "Create schedule" واختر: الجدول: يومي (Daily) — الاحتفاظ: 7 أيام — الموقع: nam5 أو me-central1
  4. تحقق من وصول النسخة الأولى
    بعد الحفظ، ستظهر النسخة الاحتياطية الأولى خلال 24 ساعة في Cloud Storage. تحقق من وصولها للتأكد من عمل الإعداد.
⚠️
بديل: تصدير يدوي عبر gcloud CLI
إذا لم تكن خطة Blaze متاحة حالياً:
gcloud firestore export gs://onway-74c20.appspot.com/backups/$(date +%Y%m%d)
